# 프로그래머의 뇌

## 1장 정리: 코딩 중 겪는 혼란

1장은 "코드를 보다가 막히는 상태"를 하나로 뭉뚱그리지 말자는 것이다.  
헷갈림에는 여러 종류가 있고, 종류에 따라 해결 방법도 달라진다.

### 메모

- 코드를 이해할 때는 장기 기억, 단기 기억, 작업 기억이 함께 쓰인다.
- 장기 기억에는 이미 알고 있는 문법, 개념, 패턴, 경험이 들어 있다.
- 단기 기억은 지금 눈앞에서 본 정보를 잠깐 붙잡는다.
- 작업 기억은 지금 보고 있는 정보와 이미 아는 지식을 조합해서 의미를 만든다.
- 그래서 코드가 안 읽히는 이유는 "내가 멍청해서"가 아니라, 어떤 기억 체계가 부담을 받고 있는지의 문제일 수 있다.

### 상세

내가 코드를 읽을 때 막히는 상황은 대략 이렇게 나눠볼 수 있다.

- 지식 부족: 개념 자체를 모르는 경우
- 정보 부족: 필요한 맥락이나 주변 코드가 없는 경우
- 처리 부담: 아는 내용인데 한 번에 머릿속에서 다루기에는 복잡한 경우

예를 들어 Redis Stream 코드를 읽는다고 하면:

- `XREADGROUP`, `XPENDING`, `XCLAIM` 자체를 모르면 지식 부족이다.
- 이 메시지가 왜 발행됐는지, consumer group이 어디서 만들어졌는지 모르면 정보 부족이다.
- 명령어는 아는데 장애 복구 흐름, 재처리, 중복 처리까지 한 번에 생각하다가 꼬이면 처리 부담이다.

## 2장 정리: 코드를 빠르게 읽는 방법

2장은 숙련자는 코드를 글자 단위로 읽지 않고, 익숙한 덩어리로 읽는다는 점이다.  
코드 읽기 속도는 눈이 빨라져서라기보다, 의미 있는 단위로 묶어보는 능력이 늘어서 좋아진다.

### 메모

- 사람은 한 번에 많은 정보를 그대로 처리하기 어렵다.
- 하지만 익숙한 패턴은 하나의 덩어리처럼 인식할 수 있다.
- 이 덩어리 인식이 코드 읽기에서 중요하다.
- 문법, API, 자주 나오는 구조가 익숙해질수록 코드가 덜 복잡하게 느껴진다.

예를 들어 아래 코드는 각각 여러 요소로 보이지만, 익숙해지면 하나의 패턴으로 읽힌다.

```csharp
await using var transaction = await db.Database.BeginTransactionAsync();
```

## 3장 정리: 문법을 빠르게 배우는 방법

3장은 문법을 매번 검색하면 코드 이해에 쓰는 에너지가 줄어든다는 것이다.  
자주 쓰는 문법과 API는 장기 기억에 들어가 있어야 코드 읽기와 문제 해결에 더 많은 작업 기억을 쓸 수 있다.

### 메모

- 문법을 외우는 것은 낡은 공부법이 아니라, 작업 기억을 아끼기 위한 기반이다.
- 모든 것을 외울 필요는 없지만, 자주 쓰는 것은 자동으로 떠올라야 한다.
- 플래시카드와 간격 반복은 문법과 개념을 장기 기억에 넣는 데 도움이 된다.
- 그냥 다시 읽는 것보다 직접 떠올리는 연습이 더 중요하다.

## 4장 정리: 복잡한 코드 읽는 방법

4장은 복잡한 코드를 읽을 때 머리가 힘든 이유를 인지 부하로 설명한다.  
코드 자체가 어려운 경우도 있지만, 표현 방식이나 내 지식 부족 때문에 더 어렵게 느껴지는 경우도 있다.

### 메모

- 프로그래밍과 관련된 인지 부하는 크게 내재적 인지 부하와 외재적 인지 부하로 나눌 수 있다.
- 내재적 인지 부하는 코드가 원래 가지고 있는 복잡성에서 온다.
- 외재적 인지 부하는 코드가 표현된 방식, 이름, 구조, 또는 읽는 사람의 지식 부족 때문에 생긴다.
- 코드를 읽는 사람의 지식 수준에 맞춰 리팩터링하면 인지 부하를 줄일 수 있다.
- 의존 그래프를 그리면 코드 요소들이 어떻게 연결되는지 보기 쉬워진다.
- 상태표를 그리면 실행 중 변수가 어떻게 바뀌는지 따라가기 쉬워진다.

### 상세

복잡한 코드를 볼 때 먼저 생각할 점은 "이 코드가 본질적으로 복잡한가, 아니면 표현이 나를 더 힘들게 만드는가"다.

예를 들어 장애 복구 코드는 원래 생각할 것이 많다.

- 메시지를 읽는다.
- 처리한다.
- 성공하면 ACK한다.
- 실패하면 Pending에 남는다.
- 다른 Consumer가 가져가도 되는지 판단한다.
- 재처리와 중복 처리를 고려한다.

이런 흐름 자체는 내재적 인지 부하다.  
도메인 문제라서 없애기 어렵다.

반면 변수 이름이 모호하거나, 한 메서드에 조회, 처리, ACK, 로깅이 모두 섞여 있거나, 실행 순서가 코드에 잘 드러나지 않는다면 외재적 인지 부하가 커진다.  
이 부분은 리팩터링으로 줄일 수 있다.

복잡한 코드를 읽을 때는 두 가지 도구가 도움이 된다.

첫 번째는 의존 그래프다.  
어떤 함수가 어떤 함수를 호출하는지, 어떤 데이터가 어디서 만들어져 어디로 넘어가는지 표시하면 구조가 보인다.

두 번째는 상태표다.  
반복문이나 조건문이 많은 코드에서는 머릿속으로만 추적하지 말고, 단계별로 변수 값이 어떻게 바뀌는지 표로 적으면 흐름이 덜 꼬인다.

## 5장 정리: 코드를 더 깊이 있게 이해하기

5장은 코드를 단순히 읽는 것과 깊이 이해하는 것은 다르다는 점을 다룬다.  
문법을 알아서 코드가 "무엇을 하는지" 읽는 것에서 더 나아가, 작성자가 "왜 이렇게 짰는지"까지 추론해야 한다.

### 메모

- 생소한 코드를 읽을 때는 변수의 역할을 파악하는 것이 깊은 이해에 도움이 된다.
- 예를 들어 어떤 변수는 반복을 진행하는 스테퍼이고, 어떤 변수는 계산이 도달해야 할 목적값일 수 있다.
- 코드 이해에는 텍스트 구조 지식과 계획 지식이 함께 필요하다.
- 텍스트 구조 지식은 코드에 사용된 문법, 표현, 제어 구조를 읽는 능력이다.
- 계획 지식은 코드 작성자의 의도와 해결 전략을 추론하는 능력이다.
- 코드를 읽는 과정은 자연언어 텍스트를 읽는 과정과 비슷한 점이 많다.
- 그래서 자연언어를 배우고 이해하는 능력은 프로그래밍을 배우는 능력과도 연결될 수 있다.
- 자연언어 텍스트를 깊이 이해할 때 쓰는 활성화, 시각화, 요약 전략은 코드 이해에도 적용할 수 있다.

### 상세

#### 변수와 코드 조각의 역할

코드를 깊이 이해하려면 먼저 변수와 코드 조각의 역할을 이름 붙여 볼 수 있어야 한다.

예를 들어 반복문 안의 `i`나 `current`가 단순히 "값이 바뀌는 변수"가 아니라, 흐름을 한 단계씩 진행시키는 스테퍼라면 코드의 구조가 다르게 보인다.  
또 어떤 변수는 최종적으로 맞춰야 하는 목적값이고, 어떤 변수는 중간 계산 결과를 잠시 담아두는 임시 저장소일 수 있다.

이렇게 역할을 붙이면 코드를 문장처럼 읽는 데서 끝나지 않고, 작성자의 계획을 추론할 수 있다.

#### 텍스트 구조 지식과 계획 지식

텍스트 구조 지식은 코드 표면을 읽는 힘이다.  
예를 들어 `for`, `if`, `return`, 함수 호출, 변수 대입이 무엇을 의미하는지 아는 것이다.

계획 지식은 그 표면 뒤의 의도를 읽는 힘이다.  
예를 들어 "왜 여기서 먼저 필터링하지?", "왜 이 값을 누적하지?", "왜 실패한 메시지를 바로 지우지 않고 Pending에 남기지?" 같은 질문에 답하려는 것이다.

#### 자연언어 읽기와 코드 읽기

자연언어 글을 깊이 읽을 때도 비슷한 일이 일어난다.  
문장 뜻만 아는 것과 글쓴이의 주장 구조를 이해하는 것은 다르다.

코드도 마찬가지다.  
문법만 읽으면 코드가 실행되는 순서는 알 수 있지만, 문제를 어떻게 나누고 어떤 전략으로 해결하려 했는지는 놓칠 수 있다.

#### 코드 이해에 적용할 독해 전략

그래서 생소한 코드를 읽을 때는 자연언어 독해 전략을 코드에도 적용해볼 수 있다.

- 활성화: 이미 알고 있는 도메인 지식이나 비슷한 코드 경험을 먼저 떠올린다.
- 시각화: 데이터 흐름, 호출 관계, 상태 변화를 그림이나 표로 그린다.
- 요약: 함수나 블록을 한 문장으로 줄여서 "결국 무엇을 하는가"를 적는다.

깊은 이해는 코드를 오래 쳐다보는 것보다, 코드에 역할과 의도를 붙여보는 과정에 가깝다.

## 6장 정리: 코딩 문제 해결을 더 잘하려면

6장은 문제를 어떻게 표현하느냐가 문제 해결 방식에 큰 영향을 준다는 점을 다룬다.  
같은 문제라도 머릿속에서 어떤 모델로 바라보는지에 따라 설계, 코드 구조, 해결 전략이 달라질 수 있다.

### 메모

- 문제를 표현하는 방법은 문제에 대한 생각에 큰 영향을 미친다.
- 예를 들어 고객을 목록으로 볼지, 집합으로 볼지에 따라 고객 객체를 저장하고 분석하는 방법이 달라질 수 있다.
- 정신 모델은 우리가 문제를 생각할 때 머릿속에 만드는 표상이다.
- 사람은 하나의 문제에 대해 서로 경쟁하는 여러 정신 모델을 가질 수 있다.
- 개념적 기계는 실제 컴퓨터가 어떻게 동작하는지를 추상적으로 표현한 모델이다.
- 개념적 기계는 프로그래밍 개념을 설명하고, 코드 동작을 추론할 때 사용된다.
- 개념적 기계는 기존 스키마타를 프로그래밍에 적용하게 해주기 때문에 프로그래밍 이해에 도움이 된다.
- 여러 개념적 기계는 서로를 보완할 수 있지만, 때로는 상충되는 정신 모델을 만들 수도 있다.

### 상세

#### 문제 표현 방식

문제를 어떻게 표현하느냐는 문제를 푸는 방식에 직접 영향을 준다.

예를 들어 "고객"을 목록으로 생각하면 순서, 추가, 삭제, 반복 같은 흐름이 먼저 떠오른다.  
반면 고객을 집합으로 생각하면 중복 제거, 포함 여부, 교집합, 차집합 같은 관계가 먼저 떠오른다.

같은 고객 데이터라도 머릿속 표현이 달라지면 저장 방식, 분석 방식, 코드 구조도 달라질 수 있다.  
그래서 코딩 문제를 풀 때는 "이 문제를 어떤 형태로 보고 있는가"를 먼저 점검하는 것이 중요하다.

#### 정신 모델

정신 모델은 문제를 이해하기 위해 머릿속에 만드는 표상이다.

코드를 작성할 때 우리는 실제 문제 자체를 그대로 다루지 않는다.  
문제에서 중요한 요소를 골라내고, 관계를 단순화해서 머릿속 모델로 만든 뒤 그 모델을 기준으로 코드를 짠다.

한 사람 안에도 여러 정신 모델이 동시에 존재할 수 있다.  
예를 들어 같은 데이터를 "순서가 있는 목록"으로 볼 수도 있고, "중복 없는 집합"으로 볼 수도 있다.

이 모델들은 서로 보완되기도 하지만, 어느 모델을 기준으로 생각하느냐에 따라 전혀 다른 해결책을 떠올리게 만들 수도 있다.

#### 개념적 기계

개념적 기계는 실제 컴퓨터가 어떻게 동작하는지를 설명하기 위해 만든 추상적인 모델이다.

예를 들어 변수는 "값을 담는 상자"처럼 설명할 수 있고, 함수 호출은 "작업을 맡기고 결과를 돌려받는 과정"처럼 설명할 수 있다.  
이런 설명은 실제 컴퓨터의 모든 세부 동작을 그대로 보여주지는 않지만, 프로그래밍 개념을 이해하고 추론하는 데 도움을 준다.

개념적 기계가 유용한 이유는 이미 알고 있는 스키마타를 프로그래밍에 연결해주기 때문이다.  
낯선 개념을 완전히 새로 외우는 대신, 이미 익숙한 비유나 구조를 통해 이해할 수 있다.

#### 여러 모델의 보완과 충돌

여러 개념적 기계는 서로를 보완할 수 있다.

예를 들어 변수는 "상자"로 이해하면 값을 저장한다는 점이 잘 보이고, "이름표"로 이해하면 어떤 값을 가리킨다는 점이 잘 보인다.  
두 모델을 함께 쓰면 변수 개념을 더 풍부하게 이해할 수 있다.

하지만 서로 다른 모델이 항상 잘 맞는 것은 아니다.  
어떤 모델은 특정 상황에서는 도움이 되지만, 다른 상황에서는 잘못된 추론을 만들 수 있다.

그래서 문제를 풀 때는 지금 쓰고 있는 정신 모델이나 개념적 기계가 이 문제에 잘 맞는지 점검해야 한다.  
막히는 순간에는 코드만 더 들여다보기보다, 문제를 표현하는 방식을 바꿔보는 것이 해결의 실마리가 될 수 있다.
