# 프로그래머의 뇌

## 1장 정리: 코딩 중 겪는 혼란

1장은 "코드를 보다가 막히는 상태"를 하나로 뭉뚱그리지 말자는 것이다.  
헷갈림에는 여러 종류가 있고, 종류에 따라 해결 방법도 달라진다.

### 메모

- 코드를 이해할 때는 장기 기억, 단기 기억, 작업 기억이 함께 쓰인다.
- 장기 기억에는 이미 알고 있는 문법, 개념, 패턴, 경험이 들어 있다.
- 단기 기억은 지금 눈앞에서 본 정보를 잠깐 붙잡는다.
- 작업 기억은 지금 보고 있는 정보와 이미 아는 지식을 조합해서 의미를 만든다.
- 그래서 코드가 안 읽히는 이유는 "내가 멍청해서"가 아니라, 어떤 기억 체계가 부담을 받고 있는지의 문제일 수 있다.

### 상세

내가 코드를 읽을 때 막히는 상황은 대략 이렇게 나눠볼 수 있다.

- 지식 부족: 개념 자체를 모르는 경우
- 정보 부족: 필요한 맥락이나 주변 코드가 없는 경우
- 처리 부담: 아는 내용인데 한 번에 머릿속에서 다루기에는 복잡한 경우

예를 들어 Redis Stream 코드를 읽는다고 하면:

- `XREADGROUP`, `XPENDING`, `XCLAIM` 자체를 모르면 지식 부족이다.
- 이 메시지가 왜 발행됐는지, consumer group이 어디서 만들어졌는지 모르면 정보 부족이다.
- 명령어는 아는데 장애 복구 흐름, 재처리, 중복 처리까지 한 번에 생각하다가 꼬이면 처리 부담이다.

## 2장 정리: 코드를 빠르게 읽는 방법

2장은 숙련자는 코드를 글자 단위로 읽지 않고, 익숙한 덩어리로 읽는다는 점이다.  
코드 읽기 속도는 눈이 빨라져서라기보다, 의미 있는 단위로 묶어보는 능력이 늘어서 좋아진다.

### 메모

- 사람은 한 번에 많은 정보를 그대로 처리하기 어렵다.
- 하지만 익숙한 패턴은 하나의 덩어리처럼 인식할 수 있다.
- 이 덩어리 인식이 코드 읽기에서 중요하다.
- 문법, API, 자주 나오는 구조가 익숙해질수록 코드가 덜 복잡하게 느껴진다.

예를 들어 아래 코드는 각각 여러 요소로 보이지만, 익숙해지면 하나의 패턴으로 읽힌다.

```csharp
await using var transaction = await db.Database.BeginTransactionAsync();
```

## 3장 정리: 문법을 빠르게 배우는 방법

3장은 문법을 매번 검색하면 코드 이해에 쓰는 에너지가 줄어든다는 것이다.  
자주 쓰는 문법과 API는 장기 기억에 들어가 있어야 코드 읽기와 문제 해결에 더 많은 작업 기억을 쓸 수 있다.

### 메모

- 문법을 외우는 것은 낡은 공부법이 아니라, 작업 기억을 아끼기 위한 기반이다.
- 모든 것을 외울 필요는 없지만, 자주 쓰는 것은 자동으로 떠올라야 한다.
- 플래시카드와 간격 반복은 문법과 개념을 장기 기억에 넣는 데 도움이 된다.
- 그냥 다시 읽는 것보다 직접 떠올리는 연습이 더 중요하다.

## 4장 정리: 복잡한 코드 읽는 방법

4장은 복잡한 코드를 읽을 때 머리가 힘든 이유를 인지 부하로 설명한다.  
코드 자체가 어려운 경우도 있지만, 표현 방식이나 내 지식 부족 때문에 더 어렵게 느껴지는 경우도 있다.

### 메모

- 프로그래밍과 관련된 인지 부하는 크게 내재적 인지 부하와 외재적 인지 부하로 나눌 수 있다.
- 내재적 인지 부하는 코드가 원래 가지고 있는 복잡성에서 온다.
- 외재적 인지 부하는 코드가 표현된 방식, 이름, 구조, 또는 읽는 사람의 지식 부족 때문에 생긴다.
- 코드를 읽는 사람의 지식 수준에 맞춰 리팩터링하면 인지 부하를 줄일 수 있다.
- 의존 그래프를 그리면 코드 요소들이 어떻게 연결되는지 보기 쉬워진다.
- 상태표를 그리면 실행 중 변수가 어떻게 바뀌는지 따라가기 쉬워진다.

### 상세

복잡한 코드를 볼 때 먼저 생각할 점은 "이 코드가 본질적으로 복잡한가, 아니면 표현이 나를 더 힘들게 만드는가"다.

예를 들어 장애 복구 코드는 원래 생각할 것이 많다.

- 메시지를 읽는다.
- 처리한다.
- 성공하면 ACK한다.
- 실패하면 Pending에 남는다.
- 다른 Consumer가 가져가도 되는지 판단한다.
- 재처리와 중복 처리를 고려한다.

이런 흐름 자체는 내재적 인지 부하다.  
도메인 문제라서 없애기 어렵다.

반면 변수 이름이 모호하거나, 한 메서드에 조회, 처리, ACK, 로깅이 모두 섞여 있거나, 실행 순서가 코드에 잘 드러나지 않는다면 외재적 인지 부하가 커진다.  
이 부분은 리팩터링으로 줄일 수 있다.

복잡한 코드를 읽을 때는 두 가지 도구가 도움이 된다.

첫 번째는 의존 그래프다.  
어떤 함수가 어떤 함수를 호출하는지, 어떤 데이터가 어디서 만들어져 어디로 넘어가는지 표시하면 구조가 보인다.

두 번째는 상태표다.  
반복문이나 조건문이 많은 코드에서는 머릿속으로만 추적하지 말고, 단계별로 변수 값이 어떻게 바뀌는지 표로 적으면 흐름이 덜 꼬인다.
